# Holiday Train Ticketing System

## 개요
명절 승차권 예매 환경을 가정한 대규모 예약 시스템 프로젝트입니다.

이 프로젝트의 목적은 대규모 트래픽 환경에서 발생하는 문제를 직접 경험하고 해결하는 과정을 학습하기 위해서입니다.

현재까지의 실무 경험은 신규 웹 서비스 및 백엔드 API 개발에 집중되어 있어서, 수만~수십만 명의 사용자가 동시에 접속하는 환경을 설계하고 운영해 본 경험이 없었습니다.

따라서 이 프로젝트를 통해 대규모 시스템 설계, 트래픽 제어, 동시성 처리, 장애 대응을 경험하고 검증하고자 합니다.


## 현재 상태

브라우저에서 예약 버튼을 누르면 대기열에 들어가고, 순번이 실시간으로 갱신되다가,
자기 차례가 오면 자동으로 입장해 로그인 화면으로 넘어갑니다. 로그인하면 예약 화면에 도착하고,
거기서부터 `reservation-ttl` 안에 예약을 마쳐야 합니다.
설계값은 3분이고, 지금은 만료를 눈으로 확인하기 쉽게 **1분**으로 낮춰 두었습니다.

- `POST /api/v1/waiting-queue` → 201, `{"token", "windowId", "seq"}`
  Redis Sorted Set + INCR 카운터. 두 연산은 Lua 스크립트로 원자적으로 묶었습니다.
- `GET /api/v1/waiting-queue/{token}?windowId=` → 200,
  `{"token", "windowId", "state", "position", "ahead", "behind", "total"}`
  `state`는 `WAITING` 또는 `ADMITTED`입니다.
  `ZRANK`와 `ZCARD`를 Lua로 묶어 한 스냅샷에서 읽습니다.
  따로 부르면 그 사이에 진입이 끼어들어 "앞 + 뒤 + 1 = 전체"가 어긋납니다.
- `POST /api/v1/waiting-queue/{token}/admission?windowId=` → 204 + `pass` 쿠키
  입장 확정. 활성 유지 시간이 `admission-grace`에서 `session-ttl`로 늘어납니다.
- **입장 제어.** 스케줄러가 1초마다 활성 정원의 빈자리를 세고 대기열 앞에서 채웁니다.
  활성 인원은 `queue.capacity`를 넘지 않습니다.
- 키는 KST 날짜 단위로 분리되고 TTL이 붙습니다.
- 랜딩 화면의 예약 버튼 → 대기 팝업(네이티브 `<dialog>`) → 5초 간격(지터 포함) 폴링 → `/login` 이동.
  팝업은 "나의 대기 순서"와 "내 뒤의 대기자 수"를 보여줍니다.
  막대는 여전히 장식입니다 — 남은 시간을 추정하려면 방출 속도를 알아야 하는데
  그 값은 아직 클라이언트에 주지 않습니다.
  대기 토큰은 JS 메모리에만 두므로 **대기 중에는 새로고침·닫기·Esc 어느 쪽이든 자리를 잃습니다.**
  입장한 뒤에는 `pass` 쿠키가 대신하므로 새로고침해도 유지됩니다.
- **입장권 게이트.** `/login`과 `/reservation`은 `pass` 쿠키가 Redis `active`에 살아 있어야 열립니다.
  쿠키가 없거나 위조됐거나 자리가 회수됐으면 랜딩으로 돌려보냅니다.
  이게 붙기 전까지는 주소창에 `/login`을 직접 치면 대기열을 통째로 건너뛸 수 있었습니다.
- **로그인.** `POST /login` — BCrypt 해시 비교, 성공하면 세션에 회원 정보를 담고 `/reservation`으로 보냅니다.
  `/reservation`은 입장권과 로그인 **둘 다** 요구합니다. 회원가입은 아직 없어 시드 계정 3개로 확인합니다.
- **예약 제한 시간.** 로그인하는 순간 활성 슬롯의 만료가 `reservation-ttl`로 다시 찍힙니다.
  그 시간이 지나면 슬롯이 회수되고, 다음 요청에서 게이트가 랜딩으로 돌려보내며 로그인 세션도 함께 끊습니다.
  예약 화면 헤더에 남은 시간이 `mm:ss`로 보이고 0에서 스스로 랜딩으로 이동하지만,
  그건 안내일 뿐입니다 — **판정은 요청마다 서버가 Redis를 보고 내리므로 타이머를 멈춰도 시간은 늘지 않습니다.**
- **로그아웃은 슬롯을 반납합니다.** 만료를 기다리지 않고 `ZREM`으로 즉시 비웁니다.
  덤으로 "로그아웃했다가 다시 로그인해 제한 시간을 새로 받는" 경로가 막힙니다 —
  슬롯이 없으면 로그인 화면 자체에 도달하지 못하고 대기열부터 다시 타야 합니다.
- **시연용 대기열 시드.** 랜딩에 `?demo`를 붙이면 더미 대기자를 밀어넣는 패널이 뜹니다.
  빈 대기열에서는 순번 1로 시작해 1초 만에 입장해 버려서, 정작 이 프로젝트의 핵심인
  "순번이 줄어드는 과정"을 화면으로 보여줄 수가 없었습니다. 아래 [시연용 시드](#시연용-시드) 참고.
- Spring Boot 4 + Java 21 가상 스레드, Redis 8 단일 노드(Docker, CPU 1코어 상한),
  PostgreSQL 18(Docker, 호스트 포트 **5433**).

활성 슬롯의 만료는 단계마다 다시 찍힙니다. 값은 하나(`active` ZSet의 score)뿐이고,
그게 곧 "이 사람이 자원을 쥐고 있는 시간"입니다.

| 단계 | 만료 | 뜻 |
|---|---|---|
| 승격(스케줄러) | `admission-grace` 150초 | 입장권을 쓸 시간 |
| 입장 확정(claim) | `session-ttl` 10분 | 로그인할 시간 |
| 로그인 | `reservation-ttl` (설계 3분, 현재 1분) | 예약할 시간 |
| 로그아웃 | 즉시 삭제 | — |

만료 판정을 위한 인터셉터를 따로 두지 않은 이유가 여기 있습니다.
기존 게이트가 이미 매 요청 이 score를 보고 있어서, 로그인 시점에 값을 짧게 다시 찍는 것만으로
"제한 시간이 지나면 예약할 수 없다"가 그대로 강제됩니다. 진실 원천이 둘로 갈라지지 않습니다.

> 그래서 다음 페이즈에서 좌석·결제 엔드포인트를 만들 때 **`WebConfig`의 화이트리스트에
> 반드시 추가해야 합니다.** 게이트를 타지 않는 경로는 제한 시간이 지나도 그대로 열려 있습니다.

아직 없는 것: 대기 중 이탈 처리, 회원가입, 좌석, WAS 다중화.

**알려진 한계 — 유령 항목.** 로그아웃이 `active`에서 빠지는 길을 열었지만,
**대기열(`waiting`) 쪽은 그대로입니다.** 대기 중에 새로고침이나 닫기로 재진입하면
이전 항목이 ZSet에 남습니다. 그 유령들이 뒤에 오는 사람들의 "내 앞 사람 수"를 부풀립니다.

입장 제어가 붙으면서 **문제가 한 겹 늘었습니다.** 유령도 순서가 되면 승격되어
활성 슬롯을 차지합니다. 아무도 쓰지 않는 자리를 `admission-grace`(150초)만큼 잡고 있는 셈이라,
유령 비율이 높을수록 실효 정원이 그만큼 깎입니다.
`admission-grace`를 짧게 잡아 회수를 앞당기는 것으로 완화만 했는데,
폴링 주기를 2초에서 5초로 올리며 같은 배율(30배)을 유지하려고 `admission-grace`도
60초에서 150초로 늘렸습니다. 즉 유령이 슬롯을 붙잡는 시간이 2.5배 길어져,
조회 부하를 줄인 대가로 이 완화 효과는 약해졌습니다.
근본 해결은 다음 페이즈의 이탈 처리입니다.

다만 **정상 사용자가 슬롯을 쥐는 시간은 짧아졌습니다.** 예약 제한이 붙기 전에는
로그인 후에도 `session-ttl`(10분)을 그대로 들고 있었는데, 이제 `reservation-ttl`이면 회수됩니다.
유령이 아니라 정원 회전 쪽의 개선이라 위 문제를 상쇄하지는 않습니다.

**알려진 한계 — CSRF 토큰이 없습니다.** Spring Security를 도입하지 않았기 때문입니다.
세션 쿠키에 `SameSite=Lax`를 걸어 크로스 사이트 POST에 쿠키가 실리지 않게 하는 것으로
대신하고 있는데, 이건 토큰의 완전한 대체가 아닙니다. 같은 사이트 안의 XSS나
`SameSite`를 무시하는 오래된 클라이언트에는 소용이 없습니다.

**알려진 한계 — 세션이 인스턴스 메모리에 있습니다.** 톰캣 기본 세션이라
WAS를 다중화하는 순간 인스턴스마다 세션이 따로 놀아 로그인이 요청마다 풀립니다.
`spring-session-data-redis`로 옮기면 되고, 바꿀 곳은 `LoginSession` 바깥에는 없습니다.
지금 넣지 않은 건 그 문제를 실제로 겪는 시점이 WAS 다중화 페이즈라서입니다.

## 인프라

Redis와 PostgreSQL 모두 `infra/docker-compose.yml`에 있습니다.

```bash
docker compose -f infra/docker-compose.yml up -d
```

`postgres`는 호스트 **5433** 포트에 붙습니다. 5432가 아닌 이유는, 개발 머신에 이미
Postgres가 깔려 있으면 그쪽이 `127.0.0.1:5432`와 `[::1]:5432`를 잡는데
Docker의 `0.0.0.0:5432`보다 구체적인 바인딩이라 `localhost` 접속이 전부 그쪽으로 가기 때문입니다.
컨테이너는 멀쩡히 떠 있는데 앱만 `role "ticketing" does not exist`로 실패하는,
원인을 찾기 유난히 성가신 형태로 드러납니다.

```bash
docker exec postgres psql -U ticketing -d ticketing -c 'SELECT login_id, name FROM member'
```

### 왜 PostgreSQL인가

계정 테이블 하나만 놓고 보면 어느 DB든 상관없습니다. 근거는 **다음 페이즈**에 있습니다.

- **배타 제약(`EXCLUDE` + GiST).** 구간 열차 예매의 핵심 동시성 문제는
  "같은 좌석의 겹치는 구간을 두 사람이 예약"입니다. Postgres는 이걸 애플리케이션 락 없이
  DB 제약으로 선언할 수 있습니다. MySQL에는 대응물이 없어 락으로 직접 풀어야 합니다.
- **MVCC.** 읽기가 쓰기를 막지 않고, 갭 락·넥스트키 락 같은 함정이 적습니다.
- `SELECT ... FOR UPDATE SKIP LOCKED`, 부분 인덱스, 네이티브 `TIMESTAMPTZ`.
- `EXPLAIN ANALYZE` 출력이 병목 분석에 훨씬 친절합니다 — 측정이 목적인 이 프로젝트에 맞습니다.

스키마는 Flyway가 기동 시점에 적용합니다(`db/migration`).
`docker-entrypoint-initdb.d`를 쓰지 않은 건 그쪽이 **볼륨이 비어 있을 때만** 돌기 때문입니다.
스키마를 고치고 다시 띄우면 아무 일도 일어나지 않은 채 앱만 깨집니다.

> Spring Boot 4는 자동 구성을 기술별 모듈로 쪼갰습니다. `flyway-core`만 넣으면
> 라이브러리는 클래스패스에 있는데 자동 구성이 붙지 않아 **마이그레이션이 조용히 실행되지 않습니다.**
> 기동은 멀쩡히 성공하고 테이블만 없는 상태가 되므로 특히 알아채기 어렵습니다.
> `org.springframework.boot:spring-boot-flyway`가 필요합니다.

## 시연용 시드

k6를 돌리지 않고 브라우저에서 대기 화면을 보려면 대기열에 사람이 있어야 합니다.
랜딩에 `?demo`를 붙이면 더미 대기자를 밀어넣는 패널이 우하단에 뜹니다.

```
http://localhost:8080/?demo
```

인원을 넣고 `채우기` → `명절 승차권` 클릭 → 팝업의 순번이 그 인원부터 시작해
초당 `queue.max-batch`(500)명씩 줄어들다가 입장합니다.

```bash
# 패널 없이 API로 직접 조작해도 됩니다
curl -X POST 'http://localhost:8080/api/v1/demo/queue?count=20000'   # → {"waiting":20000,"active":0}
curl      'http://localhost:8080/api/v1/demo/queue'                  # 현재 인원
curl -X DELETE 'http://localhost:8080/api/v1/demo/queue'             # 창 전체 초기화
```

**인원은 원하는 대기 시간 × 625로 잡습니다.** 대기열은 승격(`max-batch`, 초당 500명)과
봇 이탈(아래, 평균 초당 125명)로 빠지므로 체감 시간이 대략 `인원 ÷ 625초`입니다.
5,000명은 8초인데 폴링 주기가 5초라 순번이 한 번 찍히고 끝납니다.
기본값을 2만(약 32초)으로 둔 이유입니다.

**봇은 기다리다 포기하기도 합니다.** 승격 인원은 `promote.lua`가
`min(capacity - active, max-batch)`로 정하는데, 아래 회수 스케줄러가 정원을 계속 비워 주므로
매 주기 정확히 `max-batch`가 됩니다. 그래서 시드만 있으면 순번이 **초당 정확히 500씩** 줄어
기계처럼 보입니다. 승격 인원 계산은 프로덕션 경로라 건드리지 않고, 대신 `BotAttritionScheduler`가
1초마다 `0 ~ queue.demo.max-attrition`(기본 250)명을 대기열에서 무작위로 뺍니다.

```
max-attrition: 0            max-attrition: 250
-500  -500  -500  -500      -650  -520  -710  -660  -539
```

`ZRANDMEMBER`로 **큐 전체에서 흩어서** 뺍니다. 앞에서 순서대로 빼면 그건 이탈이 아니라 승격이고,
뒤에서만 빼면 내 순번이 줄지 않습니다. `demo-drain.lua`와 같은 `bot:` 접두사 필터를 쓰므로
**대기 중인 진짜 사용자는 이탈 대상이 아닙니다** — 무작위로 뽑는 경로라 필터가 깨지면 사용자가
소리 없이 큐에서 사라지므로, 테스트로 못 박아 두었습니다.
`max-attrition: 0`으로 두면 이탈이 꺼지고 정확히 `max-batch`씩 줄어듭니다.

**봇은 승격되는 즉시 슬롯을 반납합니다.** 이게 이 기능에서 유일하게 조심할 부분입니다.
봇도 진짜 사용자와 똑같이 `promote.lua`를 거치는데(그래야 순번이 줄어드는 속도가 사실과 같습니다),
봇은 폴링도 `claim`도 하지 않으므로 그냥 두면 아무도 쓰지 않는 활성 슬롯을
`admission-grace`(150초)만큼 붙잡습니다. 아래 [유령 항목](#남은-문제--유령-항목) 문제 그 자체입니다.
2만 명을 시드하면 정원이 봇으로 가득 차서 **진짜 사용자가 한 시간 넘게 못 들어옵니다.**
그래서 `BotDrainScheduler`가 200ms마다 `active`에서 봇만 골라 `ZREM`합니다 —
로그아웃이 슬롯을 반납하는 것과 같은 동작이라 "입장하자마자 나가는 사용자"인 셈입니다.

> 주기가 `promote-interval`(1초)보다 짧은 건 승격 직후 봇이 정원을 채운 상태로 다음 승격이
> 돌면 `room <= 0`이라 그 주기가 통째로 헛돌기 때문입니다. 대기열이 빠지는 속도가 절반이 됩니다.

알아 둘 것:

- **`전체 비우기`는 진행 중인 실사용자도 끊습니다.** `active`를 통째로 지우므로
  그 순간 로그인·예약 화면에 있던 사람은 다음 요청에서 게이트에 걸려 랜딩으로 돌아갑니다.
- **봇은 순번만 만듭니다.** 로그인도 예약도 하지 않으므로 DB·세션 쪽 부하는 재현되지 않습니다.
  그쪽까지 필요하면 k6가 맞는 도구입니다.
- **봇 이탈은 시연용 장식입니다.** 다음 페이즈의 대기열 이탈 처리를 미리 흉내 낸 것일 뿐,
  실사용자의 이탈은 여전히 처리되지 않습니다(아래 [유령 항목](#남은-문제--유령-항목)).

**규모를 키울 때.** `max-count`는 100만입니다. 실측한 비용은 아래와 같습니다.

| 인원 | 메모리 | 시드 시간 | 순번 조회 | 다 빠지는 데 |
|---|---|---|---|---|
| 10만 | 6.6MB | 0.4초 | 3~10ms | 2.7분 |
| 100만 | 66MB (상한의 26%) | 2.8초 | 3~10ms | **26분** |

메모리도 시드 속도도 조회 성능도 제약이 아닙니다. 봇 항목은 69바이트라
(`bot:12345`가 36자 UUID보다 짧아 실사용자의 115바이트보다 작습니다)
`maxmemory 256MB`의 물리적 한계는 약 380만 건이고, `active` 키와 작업 메모리를 빼면
실질 300만 건쯤입니다. 순번 조회는 `ZRANK`가 O(log N)이라 인원수를 타지 않습니다.

**걸리는 건 감소 속도뿐입니다.** 승격 500 + 이탈 평균 125 = 초당 약 625명이라
100만 명은 26분입니다. 큰 규모를 짧게 보려면 이탈을 함께 올리세요.

```bash
# 100만 명을 약 90초에 빠지게
java -jar waiting-server/build/libs/waiting-0.0.1.jar --queue.demo.max-attrition=20000
```
- 시드는 창(`windowId`) 단위입니다. KST 자정이 지나면 새 창이라 다시 채워야 합니다.
- 승차권 버튼을 **먼저** 누른 뒤 시드하면 봇이 내 뒤에 붙습니다. "내 뒤의 대기자 수"를
  채우고 싶을 때 쓰면 됩니다.

**이 기능은 `queue.demo.enabled`로 통째로 꺼집니다.** 끄면 빈이 하나도 등록되지 않아
API가 404가 되고 회수 스케줄러도 돌지 않으며 `?demo`를 붙여도 패널이 렌더링되지 않습니다.
인증이 없어 켜져 있으면 누구나 대기열을 부풀릴 수 있으므로 **외부에 노출되는 환경에서는
반드시 false여야 합니다.** `loadtest` 프로파일에서는 이미 꺼 두었습니다 —
켜 두면 회수 스케줄러가 200ms마다 `active`를 훑어서 측정 대상이 대기열이 아니라 봇 청소가 됩니다.

## 측정

부하 도구는 k6입니다. 시나리오가 둘입니다.

```bash
# 앱을 loadtest 프로파일로 띄운 뒤

# 진입: N명분 요청을 최대한 빠르게 밀어넣는다
k6 run -e USERS=100000 k6/enqueue.js

# 조회: N명이 대기하면서 5초 간격(지터 포함)으로 자기 순번을 폴링한다
k6 run -e WAITERS=5000 k6/status.js
```

두 수치를 같은 축에 놓고 비교하면 안 됩니다.
진입은 1인당 1회지만 조회는 1인당 N회라, 초당 조회 수가
`대기 인원 ÷ 폴링 주기`만큼 곱해집니다.

앱은 **IDE Run 버튼이 아니라 빌드된 jar로** 띄웁니다.
IDE는 `-XX:TieredStopAtLevel=1`을 붙이는데, C2 컴파일러를 꺼서
아무리 워밍업해도 최적화된 코드에 도달하지 못하기 때문입니다.

```bash
cd waiting-server && ./gradlew bootJar && cd ..
java -jar waiting-server/build/libs/waiting-0.0.1.jar \
     --spring.profiles.active=loadtest
```

`loadtest` 프로파일은 측정을 왜곡하는 것만 끕니다.
특히 요청당 `log.debug`를 끄는 게 중요한데, `System.out`의 락이
가상 스레드에서 pinning 지점이라 켜두면 대기열이 아니라 로거를 측정하게 됩니다.

큐에 실제로 쌓였는지, 그리고 다시 돌리기 전 초기화:

```bash
W=$(TZ=Asia/Seoul date +%Y%m%d)
docker exec redis redis-cli ZCARD waiting:holiday:$W   # 대기 인원
docker exec redis redis-cli ZCARD active:holiday:$W    # 활성 인원 (capacity를 넘으면 안 됨)

# 초기화. active 키까지 지워야 하므로 패턴이 'waiting:'이 아니라 '*:holiday:*'다.
docker exec redis redis-cli --scan --pattern '*:holiday:*' \
  | xargs -n1 docker exec redis redis-cli UNLINK
```

키 TTL이 "내일 자정 + 1시간"이라 초기화하지 않으면 같은 날 계속 누적됩니다.

정원을 바꿔 가며 보려면 실행 인자로 덮어씁니다.

```bash
java -jar waiting-server/build/libs/waiting-0.0.1.jar \
     --spring.profiles.active=loadtest \
     --queue.capacity=1000 --queue.max-batch=500 --queue.admission-grace=150s
```

### 환경

k6·애플리케이션·Redis가 **모두 같은 Mac에서** 돕니다.
부하 생성기와 측정 대상이 CPU를 나눠 쓰므로 **절대 수치는 실제 서버 성능이 아닙니다.**
목적은 절대값이 아니라 **병목이 어디서 어떤 모양으로 나타나는가**와
**변경 전후의 상대 비교**입니다.

Redis는 컨테이너 CPU 1코어, `maxmemory 256mb` + `noeviction`입니다.
대기열 항목 하나가 약 115바이트라 **약 220만 건**에서 쓰기가 거부되기 시작합니다.
버그가 아니라 설계된 동작입니다.

### 결과

2026-07-22. `loadtest` 프로파일 jar, 요청당 로그 제거,
`kern.ipc.somaxconn=1024` + `server.tomcat.accept-count=1000`.
각 런 30만 건, 매 런 전 Redis 초기화.

| 동시 실행(VUS) | 실효 rps | avg | p95 | 실패 |
|---|---|---|---|---|
| 500 | 33,073 | 14.86ms | 27.64ms | 0 |
| 1,000 | **34,129** | 28.73ms | 51.64ms | 0 |
| 2,000 | 30,912 | 61.80ms | 111.68ms | 0 |
| 4,000 | 29,096 | 131.63ms | 250.44ms | 0 |

동시 실행을 2배로 늘릴 때마다 **응답 시간은 정확히 2배가 되고 처리량은 제자리**입니다.
창구 수가 정해진 은행에 손님만 늘리는 것과 같아서, 줄이 길어질 뿐 처리량은 안 늘어납니다
(`처리량 = 동시성 ÷ 응답시간`). 이 시스템의 최적 지점은 VUS 500~1,000 근처입니다.

네 런 모두 `ZCARD`가 정확히 300,000이었습니다.
초당 3만 건씩 밀어넣는 동안 순번 유실도 중복도 없었습니다 —
`INCR`과 `ZADD`를 Lua로 원자적으로 묶은 것이 의도대로 동작했습니다.

### 입장 제어 검증

2026-07-23. `capacity=1000`, `max-batch=500`, `promote-interval=1s`, `admission-grace=60s`.
만 명을 밀어넣고 2,000명이 90초간 폴링했습니다.

- **정원을 넘지 않았습니다.** 0.5초 간격으로 `ZCARD active`를 200회 관측하는 동안
  최대치가 정확히 1,000이었습니다. `waiting + active = 10,000`도 유지됐습니다.
- **폴링 18만 건 전부 성공**했고 `앞 + 뒤 + 1 = 전체`가 한 번도 어긋나지 않았습니다.
  조회 p95는 58.51ms(임계 500ms).
- **`max-batch`가 실제로 관측됩니다.** 첫 1,000명의 입장권이 만료돼 슬롯이 한꺼번에
  회수된 순간, 대기 인원이 `10,000 → 9,500 → 9,000`으로 **두 주기에 나눠** 줄었습니다.
  상한이 없었다면 1,000명이 한 스크립트에서 처리됐을 자리입니다.

배치가 도는 순간의 지연 스파이크는 이 해상도(1초 간격 관측)로는 분리해 내지 못했습니다.
`max-batch` 값을 근거 있게 정하려면 배치 실행 시각과 요청 지연을 같은 축에 놓고 봐야 합니다.

Redis를 내리고 확인한 것 — 스케줄러가 죽지 않고 주기마다 경고 한 줄만 남기며,
Redis를 되살리면 다음 주기에 스스로 재개합니다. 진입 API는 그대로 503입니다.

> 이 런은 Docker 데몬이 꺼져 있어 **호스트의 Redis**(Homebrew, 포트 6380)로 측정했습니다.
> 위의 진입 처리량 표와 절대값을 나란히 두면 안 됩니다 — 병목의 정체인
> Docker 왕복 비용이 빠져 있습니다.

### 로그인 도입 후 회귀 확인

2026-07-23. 게이트(인터셉터)가 붙으면서 진입 경로가 느려졌는지 확인했습니다.
같은 머신에서 변경 전(HEAD) jar와 변경 후 jar를 **번갈아 3회씩** 돌렸습니다.
각 런 15만 건, VUS 1,000, 매 런 전 워밍업 300건 + Redis 초기화.

| | r1 | r2 | r3 | 평균 |
|---|---|---|---|---|
| 변경 전 | 18,378 | 18,036 | 20,712 | 19,042 |
| 변경 후 | 14,864 | 20,665 | 19,340 | 18,290 |

**측정 가능한 차이가 없습니다.** 두 계열의 산포(14.9k~20.7k)가 평균 차이(4%)보다 훨씬 큽니다.
인터셉터를 `/**` + exclude가 아니라 `/login`·`/reservation` 화이트리스트로 등록해서,
대기열 API는 인터셉터 자체를 타지 않습니다.

**다만 이 표의 절대값을 위의 34,129 rps와 나란히 두면 안 됩니다.**
변경 전 jar조차 오늘은 19k 언저리로 나왔습니다. 같은 코드가 그날 34k, 오늘 19k라면
차이는 코드가 아니라 머신 상태입니다. **비교는 같은 세션 안에서만** 유효합니다.

### 병목

애플리케이션을 빼고 `redis-benchmark`로 같은 Lua 스크립트를 직접 때려 비교했습니다
([k6/redis-baseline.sh](k6/redis-baseline.sh)).

| 측정 대상 | 처리량 |
|---|---|
| Redis 자체 (컨테이너 안) | 107,009 rps |
| 호스트 → Docker → Redis (앱과 같은 경로) | 37,908 rps |
| Spring Boot 앱 전체 | 34,129 rps |

**병목은 Spring Boot가 아니라 Docker 네트워킹입니다.**
앱의 34k는 호스트에서 Redis에 직접 접속했을 때의 한계인 38k에 거의 붙어 있고,
Spring Boot가 추가로 잡아먹은 건 10% 정도입니다.
반면 컨테이너 안팎에서 3배 가까이 깎입니다.

요청당 Docker 왕복 비용은 약 0.31ms입니다
(호스트 p50 0.399ms − 컨테이너 p50 0.087ms).
Redis가 실제로 일하는 시간은 **8.37μs**이므로,
**일하는 시간의 약 37배를 오가는 데** 쓰고 있는 셈입니다.

macOS에서 Docker가 리눅스 VM 안에서 도는 구조 때문이며 **리눅스 서버에는 없는 비용**입니다.
따라서 이 환경에서는 코드나 Redis 설정을 최적화해도 34k 위로 올라가지 않습니다.
그 위를 보려면 Redis를 리눅스 머신에 올리거나 부하 생성기를 분리해야 합니다.

### 구현하면서 정한 것

**토큰은 JS 메모리에만 둡니다**

`sessionStorage`에 넣으면 새로고침에도 살아남아 취소 규칙과 어긋납니다.
메모리에 두면 새로고침으로 자연히 사라져 규칙과 맞습니다.
대신 대기 화면을 별도 페이지로 만들 수 없습니다 — 페이지를 이동하는 순간 토큰이 날아가므로,
랜딩 위에 뜨는 팝업으로 구현했습니다.

**팝업은 네이티브 `<dialog>`입니다**

포커스 트랩, Esc 처리, 배경 딤(`::backdrop`)을 직접 만들지 않아도 됩니다.
배경 클릭으로는 안 닫히는 게 기본 동작인데, 실수로 자리를 잃는 걸 막아 주므로 그대로 뒀습니다.

닫기 버튼은 `dialog.close()`만 부르고, 정리는 `close` 이벤트 한 곳에서 합니다.
버튼에만 정리를 붙이면 **Esc로 닫았을 때 폴링이 계속 돌아** 유령 요청이 남습니다.
만료(404) 경로도 같은 곳으로 모읍니다.

`showModal()`은 첫 포커스 가능 요소를 잡기 때문에, 그냥 두면 '닫기'에 포커스 링이 걸려
닫으라고 재촉하는 화면처럼 보입니다. 숨김 제목에 초기 포커스를 줘서 링을 없앴습니다.

**앞뒤 사람 수는 한 스냅샷에서 읽습니다**

`ZRANK`가 0-based이므로 `내 앞 = ZRANK(token)`, `내 뒤 = ZCARD - ZRANK - 1`입니다.
두 명령을 따로 부르면 그 사이에 진입이 끼어들어 합이 어긋나므로,
진입 때처럼 Lua로 묶었습니다([status.lua](waiting-server/src/main/resources/redis/status.lua)).
`k6/status.js`가 매 응답마다 `앞 + 뒤 + 1 = 전체`를 검사합니다.

**windowId는 클라이언트가 돌려보냅니다**

서버 시계에서 다시 구하면, 자정 직전에 진입한 사용자가 자정 직후에 조회할 때
창이 달라져 자기 항목을 못 찾습니다. 키 TTL이 "창 마감 + 1시간"이라 그 사이 조회는 유효해야 합니다.
다만 이 값이 그대로 Redis 키가 되므로 `^\d{8}$`로 형식을 검사합니다.

**폴링은 `setInterval`이 아니라 `setTimeout` 체인입니다**

응답이 주기보다 느려지면 `setInterval`은 요청을 겹쳐 쌓습니다.
대기 인원이 많을수록 느려지는 구조라, 하필 느릴 때 부하를 스스로 키우게 됩니다.

**폴링에 지터를 더했습니다**

정각에 버튼이 열리는 시나리오에서는 수많은 사용자가 거의 같은 순간에 대기열
진입에 성공하고, 그 직후 첫 조회도 같은 위상으로 시작합니다. 이후 모두 같은
위상을 유지한 채 폴링을 반복하면 주기마다 요청이 파도처럼 몰립니다.

두 가지를 같이 했습니다. 먼저 **주기 자체를 2초에서 5초로 올렸습니다** —
지터는 몰림(분산)만 줄일 뿐 총 요청량은 그대로라, 대규모 동시 접속에서는 주기를
늘려 rps 자체를 줄이는 쪽이 더 큰 지렛대입니다. 그 위에 첫 폴링을
`0~POLL_INTERVAL_MS` 구간에서 무작위 지연 후 실행하고, 이후 매 주기에도 ±25%
지터(`POLL_JITTER_MS`)를 더해 위상이 다시 맞아들어가는 걸 막습니다. 에러
백오프도 배수 증가만 있으면 동시 장애 복구 시 재시도가 다시 몰리므로, "절반
고정 + 절반 무작위"로 흩었습니다. 순번 계산(진입 순서)에는 지터를 넣지
않았습니다 — 지터는 오직 "언제 상태를 물어보는가"에만 적용됩니다.

**클라이언트 타임아웃이 필요했습니다**

Redis를 내리고 버튼을 눌러 봤더니 서버가 503을 주는 게 아니라 그냥 매달렸습니다
(기본 프로파일의 Redis 타임아웃이 60초). 버튼이 잠긴 채 아무 안내도 안 나옵니다.
진입 10초, 조회 5초로 `AbortSignal.timeout`을 걸어 클라이언트가 먼저 끊게 했습니다.

**정원을 유지합니다. 속도를 정하지 않았습니다**

"초당 N명 방출"로 만들 수도 있었지만, 그건 유입만 늦출 뿐 활성 인원은 계속 쌓입니다.
정원을 기준으로 하면 뒷단(로그인·예약)이 동시에 상대할 인원에 상한선이 그어집니다.
대신 활성 인원의 이탈과 만료를 추적해야 해서, 나간 사람을 알아내는 일이 새로 생깁니다.

**활성 사용자를 ZSet에 담고 만료시각을 score에 넣습니다**

`active:{event}:{uuid}`처럼 키를 하나씩 두고 Redis TTL에 맡기면 만료는 공짜지만,
**인원을 세려면 `SCAN`이 필요합니다.** 정원을 유지하려면 매 주기 인원을 세야 하므로
그 비용을 낼 수 없습니다. 만료시각을 score에 넣으면 정리는 `ZREMRANGEBYSCORE` 한 번,
인원은 `ZCARD`로 정확히 나옵니다.

**승격은 스케줄러가 합니다. 조회 요청이 하지 않습니다**

조회 요청이 "내 앞에 자리가 있나"를 판정해 스스로 입장하게 만들 수도 있었습니다.
그러면 스케줄러도 리더 선출도 필요 없고, 폴링하지 않는 유령이 슬롯을 먹지도 않습니다.
그런데 만 명이 5초마다 폴링하면 약 2,000 rps인데, 그 경로에 쓰기를 섞는 대가가 큽니다.
승격을 초당 한 번 한 곳으로 모으면 조회는 읽기 전용으로 남습니다.

**입장권 유효시간이 두 단계입니다**

승격 직후에는 `admission-grace`(150초)만 주고, 사용자가 입장을 확정하면
`session-ttl`(10분)로 늘립니다. 유령은 폴링을 하지 않으니 확정까지 오지 못하고
150초 뒤 슬롯이 회수됩니다.

150초는 폴링 주기(5초)의 30배입니다. 브라우저가 비활성 탭의 타이머를 강하게 늦추기 때문에,
딴 탭에 가 있던 정상 사용자가 승격을 놓칠 여유를 준 값입니다.
짧게 잡을수록 유령 회수는 빨라지고 정상 사용자가 자리를 잃을 위험은 커집니다.

**한 주기에 올리는 인원에 상한을 뒀습니다**

Redis는 단일 스레드라 스크립트가 도는 동안 다른 모든 요청이 멈춥니다.
정원이 통째로 비는 순간 만 명을 한 번에 올리면 그게 그대로 지연 스파이크가 됩니다.
`max-batch`(500)로 잘라서 여러 주기에 나눠 올립니다.

**입장하면 대기 토큰을 쿠키로 바꿉니다**

대기 토큰은 JS 메모리에만 살아서 페이지를 옮기는 순간 사라집니다.
입장 확정 응답이 `pass` 쿠키(HttpOnly)를 심어 로그인·예약 화면까지 자격이 따라가게 합니다.
"새로고침하면 자리를 잃는다"는 규칙은 **대기 중에만** 적용됩니다 —
입장한 뒤에도 새로고침으로 자리를 잃는다면 그게 더 이상한 동작입니다.

쿠키에는 `Secure`를 붙이지 않았습니다. 로컬이 http라 붙이면 저장 자체가 안 됩니다.
HTTPS로 올리는 순간 반드시 켜야 합니다.

**Spring Security를 넣지 않고 BCrypt만 가져왔습니다**

`spring-security-crypto` 하나만 넣으면 필터 체인 없이 `BCryptPasswordEncoder`만 씁니다.
스타터를 넣으면 `permitAll`이어도 모든 요청이 필터 체인을 통과하는데,
이 프로젝트는 진입 처리량 수치의 변화를 근거로 삼기 때문에
**대기열 경로에 측정 변수를 하나 더 얹는 대가**가 큽니다.

인터셉터를 `/**` + exclude가 아니라 `/login`·`/reservation` 화이트리스트로 등록한 것도 같은 이유입니다.
대가는 새 화면을 만들 때마다 등록을 빠뜨리면 그대로 뚫린다는 점입니다.

**게이트는 쿠키의 존재가 아니라 Redis를 봅니다**

쿠키는 클라이언트가 들고 있는 값이라 얼마든지 지어낼 수 있습니다.
매 진입마다 `active` ZSet에 그 토큰이 살아 있는지 확인해야 정원과 만료가 실제로 강제됩니다.
`status.lua`가 이미 `ZSCORE`로 만료까지 판정하고 있어서 새 스크립트 없이 그대로 씁니다.
화면 진입당 한 번이라 폴링 경로와 달리 인원수만큼 곱해지지 않습니다.

그래서 **입장권 만료가 로그인 세션을 이깁니다.** 로그인한 채로 활성 자리가 회수되면
`/reservation`은 랜딩으로 돌려보냅니다. 정원 회수가 로그인 유지보다 우선입니다.

**세션 ID를 URL에 싣지 않습니다**

톰캣 기본값(`cookie,url`)이면 아직 쿠키를 돌려주지 않은 클라이언트에게
URL에 `;jsessionid=...`를 붙여 줍니다. 로그인 직후 리다이렉트가 딱 그 경우라,
세션 ID가 `Location` 헤더에 실려 브라우저 기록·Referer·접근 로그에 남습니다.
세션 ID는 곧 로그인 자격이므로 `tracking-modes: cookie`로 못 박았습니다.

세션 고정 방어(`changeSessionId()`)에는 함정이 하나 있습니다.
세션이 없을 때 부르면 `IllegalStateException`이 납니다.
이 앱은 로그인 전에 세션을 만드는 경로가 없어서, 검사 없이 부르면
**정상 로그인이 항상** 500으로 끝납니다.

### 남은 문제 — 유령 항목

대기열에서 빼는 수단이 없어서, 새로고침으로 재진입하면 이전 ZSet 항목이 남습니다.
그 유령들이 뒤에 오는 사람들의 "내 앞 사람 수"를 부풀립니다.
30만 건 테스트에서 `ZCARD`가 정확히 맞았던 만큼, 틀어지면 바로 드러납니다.

입장 제어가 붙으면서 대가가 하나 늘었습니다. 유령도 순서가 되면 승격되어
아무도 쓰지 않는 활성 슬롯을 `admission-grace`만큼 잡습니다.
순번이 부풀 뿐 아니라 **실효 정원까지 깎입니다.**

선택지는 대략 셋입니다.

- `DELETE /api/v1/waiting-queue/{token}` + 이탈 시 `sendBeacon` 호출
  — 확실하지만 beacon이 항상 도착한다는 보장은 없음
- 항목별 TTL + 주기적 하트비트 — 유실에 강하지만 조회마다 비용
- 유령을 그냥 두고 순번을 "최대치"로 안내 — 가장 단순, 정확도는 포기

폴링이 이미 5초마다 오고 있으므로, 그걸 하트비트로 재해석하는 두 번째 선택지가
새 요청을 만들지 않는다는 점에서 유리해 보입니다. 다만 조회 경로에 쓰기가 생기는데,
그건 승격을 스케줄러에 맡기며 피하려던 바로 그 비용입니다.
