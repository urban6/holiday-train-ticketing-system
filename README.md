# Holiday Train Ticketing System

## 개요
명절 승차권 예매 환경을 가정한 대규모 예약 시스템 프로젝트입니다.

이 프로젝트의 목적은 대규모 트래픽 환경에서 발생하는 문제를 직접 경험하고 해결하는 과정을 학습하기 위해서입니다.

현재까지의 실무 경험은 신규 웹 서비스 및 백엔드 API 개발에 집중되어 있어서, 수만~수십만 명의 사용자가 동시에 접속하는 환경을 설계하고 운영해 본 경험이 없었습니다.

따라서 이 프로젝트를 통해 대규모 시스템 설계, 트래픽 제어, 동시성 처리, 장애 대응을 경험하고 검증하고자 합니다.


## 현재 상태

브라우저에서 예약 버튼을 누르면 대기열에 들어가고, 순번이 실시간으로 갱신되다가,
자기 차례가 오면 자동으로 입장해 다음 화면으로 넘어갑니다.

- `POST /api/v1/waiting-queue` → 201, `{"token", "windowId", "seq"}`
  Redis Sorted Set + INCR 카운터. 두 연산은 Lua 스크립트로 원자적으로 묶었습니다.
- `GET /api/v1/waiting-queue/{token}?windowId=` → 200,
  `{"token", "windowId", "state", "position", "ahead", "behind", "total"}`
  `state`는 `WAITING` 또는 `ADMITTED`입니다.
  `ZRANK`와 `ZCARD`를 Lua로 묶어 한 스냅샷에서 읽습니다.
  따로 부르면 그 사이에 진입이 끼어들어 "앞 + 뒤 + 1 = 전체"가 어긋납니다.
- `POST /api/v1/waiting-queue/{token}/admission?windowId=` → 204 + `pass` 쿠키
  입장 확정. 활성 유지 시간이 `admission-grace`에서 `session-ttl`로 늘어납니다.
- **입장 제어.** 스케줄러가 1초마다 활성 정원의 빈자리를 세고 대기열 앞에서 채웁니다.
  활성 인원은 `queue.capacity`를 넘지 않습니다.
- 키는 KST 날짜 단위로 분리되고 TTL이 붙습니다.
- 랜딩 화면의 예약 버튼 → 대기 팝업(네이티브 `<dialog>`) → 2초 간격 폴링 → `/login` 이동.
  팝업은 "나의 대기 순서"와 "내 뒤의 대기자 수"를 보여줍니다.
  막대는 여전히 장식입니다 — 남은 시간을 추정하려면 방출 속도를 알아야 하는데
  그 값은 아직 클라이언트에 주지 않습니다.
  대기 토큰은 JS 메모리에만 두므로 **대기 중에는 새로고침·닫기·Esc 어느 쪽이든 자리를 잃습니다.**
  입장한 뒤에는 `pass` 쿠키가 대신하므로 새로고침해도 유지됩니다.
- Spring Boot 4 + Java 21 가상 스레드, Redis 8 단일 노드(Docker, CPU 1코어 상한).

아직 없는 것: 이탈 처리, 로그인(화면은 자리표시자), 좌석, DB, WAS 다중화.

**알려진 한계 — 유령 항목.** 대기열에서 빼는 수단(`ZREM`)이 없어서,
새로고침이나 닫기로 재진입하면 이전 항목이 ZSet에 그대로 남습니다.
그 유령들이 뒤에 오는 사람들의 "내 앞 사람 수"를 부풀립니다.

입장 제어가 붙으면서 **문제가 한 겹 늘었습니다.** 유령도 순서가 되면 승격되어
활성 슬롯을 차지합니다. 아무도 쓰지 않는 자리를 `admission-grace`(60초)만큼 잡고 있는 셈이라,
유령 비율이 높을수록 실효 정원이 그만큼 깎입니다.
`admission-grace`를 짧게 잡아 회수를 앞당기는 것으로 완화만 했습니다.
근본 해결은 다음 페이즈의 이탈 처리입니다.

## 측정

부하 도구는 k6입니다. 시나리오가 둘입니다.

```bash
# 앱을 loadtest 프로파일로 띄운 뒤

# 진입: N명분 요청을 최대한 빠르게 밀어넣는다
k6 run -e USERS=100000 k6/enqueue.js

# 조회: N명이 대기하면서 2초마다 자기 순번을 폴링한다
k6 run -e WAITERS=5000 k6/status.js
```

두 수치를 같은 축에 놓고 비교하면 안 됩니다.
진입은 1인당 1회지만 조회는 1인당 N회라, 초당 조회 수가
`대기 인원 ÷ 폴링 주기`만큼 곱해집니다.

앱은 **IDE Run 버튼이 아니라 빌드된 jar로** 띄웁니다.
IDE는 `-XX:TieredStopAtLevel=1`을 붙이는데, C2 컴파일러를 꺼서
아무리 워밍업해도 최적화된 코드에 도달하지 못하기 때문입니다.

```bash
cd waiting-server && ./gradlew bootJar && cd ..
java -jar waiting-server/build/libs/waiting-0.0.1.jar \
     --spring.profiles.active=loadtest
```

`loadtest` 프로파일은 측정을 왜곡하는 것만 끕니다.
특히 요청당 `log.debug`를 끄는 게 중요한데, `System.out`의 락이
가상 스레드에서 pinning 지점이라 켜두면 대기열이 아니라 로거를 측정하게 됩니다.

큐에 실제로 쌓였는지, 그리고 다시 돌리기 전 초기화:

```bash
W=$(TZ=Asia/Seoul date +%Y%m%d)
docker exec redis redis-cli ZCARD waiting:holiday:$W   # 대기 인원
docker exec redis redis-cli ZCARD active:holiday:$W    # 활성 인원 (capacity를 넘으면 안 됨)

# 초기화. active 키까지 지워야 하므로 패턴이 'waiting:'이 아니라 '*:holiday:*'다.
docker exec redis redis-cli --scan --pattern '*:holiday:*' \
  | xargs -n1 docker exec redis redis-cli UNLINK
```

키 TTL이 "내일 자정 + 1시간"이라 초기화하지 않으면 같은 날 계속 누적됩니다.

정원을 바꿔 가며 보려면 실행 인자로 덮어씁니다.

```bash
java -jar waiting-server/build/libs/waiting-0.0.1.jar \
     --spring.profiles.active=loadtest \
     --queue.capacity=1000 --queue.max-batch=500 --queue.admission-grace=60s
```

### 환경

k6·애플리케이션·Redis가 **모두 같은 Mac에서** 돕니다.
부하 생성기와 측정 대상이 CPU를 나눠 쓰므로 **절대 수치는 실제 서버 성능이 아닙니다.**
목적은 절대값이 아니라 **병목이 어디서 어떤 모양으로 나타나는가**와
**변경 전후의 상대 비교**입니다.

Redis는 컨테이너 CPU 1코어, `maxmemory 256mb` + `noeviction`입니다.
대기열 항목 하나가 약 115바이트라 **약 220만 건**에서 쓰기가 거부되기 시작합니다.
버그가 아니라 설계된 동작입니다.

### 결과

2026-07-22. `loadtest` 프로파일 jar, 요청당 로그 제거,
`kern.ipc.somaxconn=1024` + `server.tomcat.accept-count=1000`.
각 런 30만 건, 매 런 전 Redis 초기화.

| 동시 실행(VUS) | 실효 rps | avg | p95 | 실패 |
|---|---|---|---|---|
| 500 | 33,073 | 14.86ms | 27.64ms | 0 |
| 1,000 | **34,129** | 28.73ms | 51.64ms | 0 |
| 2,000 | 30,912 | 61.80ms | 111.68ms | 0 |
| 4,000 | 29,096 | 131.63ms | 250.44ms | 0 |

동시 실행을 2배로 늘릴 때마다 **응답 시간은 정확히 2배가 되고 처리량은 제자리**입니다.
창구 수가 정해진 은행에 손님만 늘리는 것과 같아서, 줄이 길어질 뿐 처리량은 안 늘어납니다
(`처리량 = 동시성 ÷ 응답시간`). 이 시스템의 최적 지점은 VUS 500~1,000 근처입니다.

네 런 모두 `ZCARD`가 정확히 300,000이었습니다.
초당 3만 건씩 밀어넣는 동안 순번 유실도 중복도 없었습니다 —
`INCR`과 `ZADD`를 Lua로 원자적으로 묶은 것이 의도대로 동작했습니다.

### 입장 제어 검증

2026-07-23. `capacity=1000`, `max-batch=500`, `promote-interval=1s`, `admission-grace=60s`.
만 명을 밀어넣고 2,000명이 90초간 폴링했습니다.

- **정원을 넘지 않았습니다.** 0.5초 간격으로 `ZCARD active`를 200회 관측하는 동안
  최대치가 정확히 1,000이었습니다. `waiting + active = 10,000`도 유지됐습니다.
- **폴링 18만 건 전부 성공**했고 `앞 + 뒤 + 1 = 전체`가 한 번도 어긋나지 않았습니다.
  조회 p95는 58.51ms(임계 500ms).
- **`max-batch`가 실제로 관측됩니다.** 첫 1,000명의 입장권이 만료돼 슬롯이 한꺼번에
  회수된 순간, 대기 인원이 `10,000 → 9,500 → 9,000`으로 **두 주기에 나눠** 줄었습니다.
  상한이 없었다면 1,000명이 한 스크립트에서 처리됐을 자리입니다.

배치가 도는 순간의 지연 스파이크는 이 해상도(1초 간격 관측)로는 분리해 내지 못했습니다.
`max-batch` 값을 근거 있게 정하려면 배치 실행 시각과 요청 지연을 같은 축에 놓고 봐야 합니다.

Redis를 내리고 확인한 것 — 스케줄러가 죽지 않고 주기마다 경고 한 줄만 남기며,
Redis를 되살리면 다음 주기에 스스로 재개합니다. 진입 API는 그대로 503입니다.

> 이 런은 Docker 데몬이 꺼져 있어 **호스트의 Redis**(Homebrew, 포트 6380)로 측정했습니다.
> 위의 진입 처리량 표와 절대값을 나란히 두면 안 됩니다 — 병목의 정체인
> Docker 왕복 비용이 빠져 있습니다.

### 병목

애플리케이션을 빼고 `redis-benchmark`로 같은 Lua 스크립트를 직접 때려 비교했습니다
([k6/redis-baseline.sh](k6/redis-baseline.sh)).

| 측정 대상 | 처리량 |
|---|---|
| Redis 자체 (컨테이너 안) | 107,009 rps |
| 호스트 → Docker → Redis (앱과 같은 경로) | 37,908 rps |
| Spring Boot 앱 전체 | 34,129 rps |

**병목은 Spring Boot가 아니라 Docker 네트워킹입니다.**
앱의 34k는 호스트에서 Redis에 직접 접속했을 때의 한계인 38k에 거의 붙어 있고,
Spring Boot가 추가로 잡아먹은 건 10% 정도입니다.
반면 컨테이너 안팎에서 3배 가까이 깎입니다.

요청당 Docker 왕복 비용은 약 0.31ms입니다
(호스트 p50 0.399ms − 컨테이너 p50 0.087ms).
Redis가 실제로 일하는 시간은 **8.37μs**이므로,
**일하는 시간의 약 37배를 오가는 데** 쓰고 있는 셈입니다.

macOS에서 Docker가 리눅스 VM 안에서 도는 구조 때문이며 **리눅스 서버에는 없는 비용**입니다.
따라서 이 환경에서는 코드나 Redis 설정을 최적화해도 34k 위로 올라가지 않습니다.
그 위를 보려면 Redis를 리눅스 머신에 올리거나 부하 생성기를 분리해야 합니다.

## 다음 페이즈

지금까지는 API 한 개를 얼마나 밀어붙일 수 있는지만 봤습니다.
다음은 **사용자가 실제로 겪는 흐름**을 만듭니다.

1. 초기 화면에서 "승차권 예약" 버튼을 클릭하면 대기열 입장
2. 대기열에 입장하면 내 앞에 남은 사람 수와 내 뒤에 대기하는 사람 수 조회 가능
   - 새로고침을 하면 대기열이 취소되고 다시 예약 버튼을 클릭해야 함
3. 대기열에서 나왔을 때, 로그인 화면이 보이고 사용자는 로그인을 할 수 있다
4. 로그인이 성공하면 예약 페이지가 나온다 — **이 기능부터는 추후 페이즈에 구현 예정**

### 진행 상황

| 요구사항 | 상태 |
|---|---|
| 1. 버튼 클릭 → 대기열 입장 | 완료 |
| 2. 앞뒤 사람 수 조회 | 완료 (`GET /api/v1/waiting-queue/{token}`, 2초 폴링) |
| 2. 새로고침 시 취소 | **절반.** 브라우저는 자리를 잃지만 ZSet 항목은 남습니다 |
| 3. 대기열에서 나옴 | 완료. 정원이 비면 스케줄러가 승격하고, 폴링이 그걸 보고 `/login`으로 넘깁니다 |
| 3. 로그인 | **자리표시자.** 화면은 뜨지만 인증도 사용자 저장소도 없고 DB 자체가 없습니다 |

### 구현하면서 정한 것

**토큰은 JS 메모리에만 둡니다**

`sessionStorage`에 넣으면 새로고침에도 살아남아 취소 규칙과 어긋납니다.
메모리에 두면 새로고침으로 자연히 사라져 규칙과 맞습니다.
대신 대기 화면을 별도 페이지로 만들 수 없습니다 — 페이지를 이동하는 순간 토큰이 날아가므로,
랜딩 위에 뜨는 팝업으로 구현했습니다.

**팝업은 네이티브 `<dialog>`입니다**

포커스 트랩, Esc 처리, 배경 딤(`::backdrop`)을 직접 만들지 않아도 됩니다.
배경 클릭으로는 안 닫히는 게 기본 동작인데, 실수로 자리를 잃는 걸 막아 주므로 그대로 뒀습니다.

닫기 버튼은 `dialog.close()`만 부르고, 정리는 `close` 이벤트 한 곳에서 합니다.
버튼에만 정리를 붙이면 **Esc로 닫았을 때 폴링이 계속 돌아** 유령 요청이 남습니다.
만료(404) 경로도 같은 곳으로 모읍니다.

`showModal()`은 첫 포커스 가능 요소를 잡기 때문에, 그냥 두면 '닫기'에 포커스 링이 걸려
닫으라고 재촉하는 화면처럼 보입니다. 숨김 제목에 초기 포커스를 줘서 링을 없앴습니다.

**앞뒤 사람 수는 한 스냅샷에서 읽습니다**

`ZRANK`가 0-based이므로 `내 앞 = ZRANK(token)`, `내 뒤 = ZCARD - ZRANK - 1`입니다.
두 명령을 따로 부르면 그 사이에 진입이 끼어들어 합이 어긋나므로,
진입 때처럼 Lua로 묶었습니다([status.lua](waiting-server/src/main/resources/redis/status.lua)).
`k6/status.js`가 매 응답마다 `앞 + 뒤 + 1 = 전체`를 검사합니다.

**windowId는 클라이언트가 돌려보냅니다**

서버 시계에서 다시 구하면, 자정 직전에 진입한 사용자가 자정 직후에 조회할 때
창이 달라져 자기 항목을 못 찾습니다. 키 TTL이 "창 마감 + 1시간"이라 그 사이 조회는 유효해야 합니다.
다만 이 값이 그대로 Redis 키가 되므로 `^\d{8}$`로 형식을 검사합니다.

**폴링은 `setInterval`이 아니라 `setTimeout` 체인입니다**

응답이 주기보다 느려지면 `setInterval`은 요청을 겹쳐 쌓습니다.
대기 인원이 많을수록 느려지는 구조라, 하필 느릴 때 부하를 스스로 키우게 됩니다.

**클라이언트 타임아웃이 필요했습니다**

Redis를 내리고 버튼을 눌러 봤더니 서버가 503을 주는 게 아니라 그냥 매달렸습니다
(기본 프로파일의 Redis 타임아웃이 60초). 버튼이 잠긴 채 아무 안내도 안 나옵니다.
진입 10초, 조회 5초로 `AbortSignal.timeout`을 걸어 클라이언트가 먼저 끊게 했습니다.

**정원을 유지합니다. 속도를 정하지 않았습니다**

"초당 N명 방출"로 만들 수도 있었지만, 그건 유입만 늦출 뿐 활성 인원은 계속 쌓입니다.
정원을 기준으로 하면 뒷단(로그인·예약)이 동시에 상대할 인원에 상한선이 그어집니다.
대신 활성 인원의 이탈과 만료를 추적해야 해서, 나간 사람을 알아내는 일이 새로 생깁니다.

**활성 사용자를 ZSet에 담고 만료시각을 score에 넣습니다**

`active:{event}:{uuid}`처럼 키를 하나씩 두고 Redis TTL에 맡기면 만료는 공짜지만,
**인원을 세려면 `SCAN`이 필요합니다.** 정원을 유지하려면 매 주기 인원을 세야 하므로
그 비용을 낼 수 없습니다. 만료시각을 score에 넣으면 정리는 `ZREMRANGEBYSCORE` 한 번,
인원은 `ZCARD`로 정확히 나옵니다.

**승격은 스케줄러가 합니다. 조회 요청이 하지 않습니다**

조회 요청이 "내 앞에 자리가 있나"를 판정해 스스로 입장하게 만들 수도 있었습니다.
그러면 스케줄러도 리더 선출도 필요 없고, 폴링하지 않는 유령이 슬롯을 먹지도 않습니다.
그런데 만 명이 2초마다 폴링하면 약 5,000 rps인데, 그 경로에 쓰기를 섞는 대가가 큽니다.
승격을 초당 한 번 한 곳으로 모으면 조회는 읽기 전용으로 남습니다.

**입장권 유효시간이 두 단계입니다**

승격 직후에는 `admission-grace`(60초)만 주고, 사용자가 입장을 확정하면
`session-ttl`(10분)로 늘립니다. 유령은 폴링을 하지 않으니 확정까지 오지 못하고
60초 뒤 슬롯이 회수됩니다.

60초는 폴링 주기의 30배입니다. 브라우저가 비활성 탭의 타이머를 강하게 늦추기 때문에,
딴 탭에 가 있던 정상 사용자가 승격을 놓칠 여유를 준 값입니다.
짧게 잡을수록 유령 회수는 빨라지고 정상 사용자가 자리를 잃을 위험은 커집니다.

**한 주기에 올리는 인원에 상한을 뒀습니다**

Redis는 단일 스레드라 스크립트가 도는 동안 다른 모든 요청이 멈춥니다.
정원이 통째로 비는 순간 만 명을 한 번에 올리면 그게 그대로 지연 스파이크가 됩니다.
`max-batch`(500)로 잘라서 여러 주기에 나눠 올립니다.

**입장하면 대기 토큰을 쿠키로 바꿉니다**

대기 토큰은 JS 메모리에만 살아서 페이지를 옮기는 순간 사라집니다.
입장 확정 응답이 `pass` 쿠키(HttpOnly)를 심어 로그인·예약 화면까지 자격이 따라가게 합니다.
"새로고침하면 자리를 잃는다"는 규칙은 **대기 중에만** 적용됩니다 —
입장한 뒤에도 새로고침으로 자리를 잃는다면 그게 더 이상한 동작입니다.

쿠키에는 `Secure`를 붙이지 않았습니다. 로컬이 http라 붙이면 저장 자체가 안 됩니다.
HTTPS로 올리는 순간 반드시 켜야 합니다.

### 남은 문제 — 유령 항목

대기열에서 빼는 수단이 없어서, 새로고침으로 재진입하면 이전 ZSet 항목이 남습니다.
그 유령들이 뒤에 오는 사람들의 "내 앞 사람 수"를 부풀립니다.
30만 건 테스트에서 `ZCARD`가 정확히 맞았던 만큼, 틀어지면 바로 드러납니다.

입장 제어가 붙으면서 대가가 하나 늘었습니다. 유령도 순서가 되면 승격되어
아무도 쓰지 않는 활성 슬롯을 `admission-grace`만큼 잡습니다.
순번이 부풀 뿐 아니라 **실효 정원까지 깎입니다.**

선택지는 대략 셋입니다.

- `DELETE /api/v1/waiting-queue/{token}` + 이탈 시 `sendBeacon` 호출
  — 확실하지만 beacon이 항상 도착한다는 보장은 없음
- 항목별 TTL + 주기적 하트비트 — 유실에 강하지만 조회마다 비용
- 유령을 그냥 두고 순번을 "최대치"로 안내 — 가장 단순, 정확도는 포기

폴링이 이미 2초마다 오고 있으므로, 그걸 하트비트로 재해석하는 두 번째 선택지가
새 요청을 만들지 않는다는 점에서 유리해 보입니다. 다만 조회 경로에 쓰기가 생기는데,
그건 승격을 스케줄러에 맡기며 피하려던 바로 그 비용입니다.
