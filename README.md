# 명절 승차권 예매

> 명절 예매급 순간 트래픽을 대기열 기반 입장 제어로 감당하는 대규모 시스템 — 병목을 직접 측정하며 설계.
> Spring Boot 4 · Java 21 · Redis 8 · PostgreSQL 18 · k6

## 프로젝트 배경 및 목적


명절 승차권 예매 환경을 가정한 대규모 예약 시스템입니다.

동시 접속자가 급증하는 상황에서 대기열, 예약 처리, 데이터베이스, 캐시 구조가 시스템 성능에 어떤 영향을 주는지 확인하기 위해서 개발을 했습니다.

단순 기능 구현보다 병목 구간을 찾고, 아키텍처 변경 전후의 처리량(TPS), 응답 시간, 자원 사용량을 측정하고 비교하 것을 목표로 합니다.

## 문제 정의
명절 승차권 예매 서비스는 특정 시각에 대규모 사용자가 동시에 접속하는 특성을 가진 서비스입니다.

평일 승차권 예매 시스템과 달리 사용자가 좌석을 선택하거나 결제를 진행하지 않고 열차 단위로 예약을 수행합니다. 이는 제한된 시간 내에 최대한 많은 예약 요청을 처리해야 하는 명절 예매의 특성상 좌석 선택 및 결제 과정에서 발생하는 추가적인 동시성 문제와 처리 지연을 최소화하기 위한 설계라고 판단했습니다.

그럼에도 불구하고 수많은 사용자가 동시에 예약을 시도하기 때문에 다음과 같은 문제가 발생합니다.

- 순간적인 트래픽 증가로 인한 서버 및 DB 과부하
- 동일 열차에 대한 예약 요청 처리
- 잔여 좌석 수량 관리 및 데이터 정합성 보장
- 장애 발생 시 예약 데이터의 일관성 유지

실제 명절 승차권 예매 환경을 기반으로 이러한 문제를 제현하고 대기열, 캐시, 동시성 제어 전략이 시스템 성능과 안정성에 미치는 영향을 검증하는 것을 목표로 합니다.

## 설계

### 대기열은 왜 Redis인가 (vs Kafka)

대기열이 매 순간 답해야 할 질문은 **"이 사용자가 지금 몇 번째인가"**입니다.
이 한 문장이 자료구조를 결정합니다. 필요한 연산은 다섯입니다 — ① 순번 조회(내 앞 몇 명)
② FIFO 순서 ③ 정원 기반 입장 제어(활성 인원 세기) ④ 항목별 만료 ⑤ 중간 이탈 제거.

- **Redis Sorted Set은 이 다섯을 자료구조 하나로 만족합니다.** 순번은 `ZRANK`가 O(log N),
  정원은 `ZCARD`로 정확히, 만료는 score에 만료시각을 넣어 `ZREMRANGEBYSCORE` 한 번, 이탈은 `ZREM`.
  여러 명령은 Lua로 묶어 "세고 · 뽑고 · 넣기"를 원자적으로 강제합니다.
- **Kafka는 append-only 커밋 로그라 "내 순번" 같은 임의 접근 조회를 못 합니다.**
  컨슈머 오프셋은 "어디까지 소비했나"이지 "몇 번째인가"가 아니고, 중간 항목 삭제(이탈)나
  항목별 만료도 없습니다. 대기열 위치 조회라는 지배적 워크로드에 자료 모델이 맞지 않습니다.
- **다만 Kafka는 뒷단 주문 파이프라인엔 맞습니다.** 확정된 예약·결제 이벤트를 durable하게
  순서대로 처리하는 곳이죠. 그래서 선택은 문제별로 갈립니다 — **대기·입장 = 정렬 집합(Redis),
  주문 처리 = durable 로그(Kafka).** 지금 단계는 앞쪽이라 Redis입니다.

### Redis를 고른 뒤 따라온 결정들

- **진실 원천은 하나 — 만료를 score에.** 개별 TTL 키 + `SCAN` 대신, `active` ZSet score에 만료시각을 넣어 정리는 `ZREMRANGEBYSCORE`, 인원은 `ZCARD`로.
- **정원 기준 입장 제어(vs 속도 기준).** 속도 제한은 유입만 늦추고 활성 인원은 쌓입니다. 정원 기준이라야 뒷단(로그인·예약)이 동시에 상대할 인원에 상한이 그어집니다.
- **승격은 스케줄러가, 조회는 읽기 전용.** 만 명이 5초마다 폴링하면 ≈2,000 rps. 그 경로에 쓰기를 섞지 않으려 승격을 초당 한 번 한 곳으로 모읍니다.
- **Redis 단일 스레드 → Lua 원자성 + `max-batch`.** 다중 명령을 원자적으로 묶고, 정원이 통째로 비는 순간의 지연 스파이크를 `max-batch`로 잘라 여러 주기에 나눕니다.
- **게이트는 쿠키가 아니라 Redis를 본다 / 입장권 2단계 TTL.** 쿠키는 위조 가능하므로 매 요청 `active`를 확인합니다. 그래서 입장권 만료가 로그인 세션을 이깁니다.
- **(다음 페이즈) 왜 PostgreSQL.** 구간 예약의 겹침 방지를 애플리케이션 락 없이 DB 제약(`EXCLUDE` + GiST)으로. → [상세](docs/design.md#왜-postgresql인가)

## 동작 흐름

예약 버튼 → 대기열 입장 → 순번 실시간 갱신 → 자기 차례에 자동 입장 → 로그인 → 예약 화면.
로그인하는 순간부터 `reservation-ttl`(설계 3분) 안에 예약을 마쳐야 합니다.

| 엔드포인트 | 동작 |
|---|---|
| `POST /api/v1/waiting-queue` | 대기열 진입 → `201 {token, windowId, seq}` |
| `GET /api/v1/waiting-queue/{token}` | 순번 조회 → `{state, position, ahead, behind, total}` |
| `POST /api/v1/waiting-queue/{token}/admission` | 입장 확정 → `204` + `pass` 쿠키 |
| `POST /login` | BCrypt 인증 → 세션 + `/reservation` |

입장 제어 스케줄러가 1초마다 정원 빈자리를 대기열 앞에서 채우고, 활성 인원은 `queue.capacity`를 넘지 않습니다.
게이트(`/login`·`/reservation`)는 쿠키 존재가 아니라 매 요청 Redis `active`를 확인해, 자리가 회수됐으면 랜딩으로 돌려보냅니다.
