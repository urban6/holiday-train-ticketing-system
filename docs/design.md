# 설계 상세 — 구현하면서 정한 것

> [← README](../README.md)

각 결정의 배경과 트레이드오프입니다. README의 [핵심 설계 결정](../README.md#핵심-설계-결정)에서 요약을 봤다면, 여기서 근거를 확인하세요.

## 왜 PostgreSQL인가

계정 테이블 하나만 놓고 보면 어느 DB든 상관없습니다. 근거는 **다음 페이즈**에 있습니다.

- **배타 제약(`EXCLUDE` + GiST).** 구간 열차 예매의 핵심 동시성 문제는
  "같은 좌석의 겹치는 구간을 두 사람이 예약"입니다. Postgres는 이걸 애플리케이션 락 없이
  DB 제약으로 선언할 수 있습니다. MySQL에는 대응물이 없어 락으로 직접 풀어야 합니다.
- **MVCC.** 읽기가 쓰기를 막지 않고, 갭 락·넥스트키 락 같은 함정이 적습니다.
- `SELECT ... FOR UPDATE SKIP LOCKED`, 부분 인덱스, 네이티브 `TIMESTAMPTZ`.
- `EXPLAIN ANALYZE` 출력이 병목 분석에 훨씬 친절합니다 — 측정이 목적인 이 프로젝트에 맞습니다.

스키마는 Flyway가 기동 시점에 적용합니다(`db/migration`).
`docker-entrypoint-initdb.d`를 쓰지 않은 건 그쪽이 **볼륨이 비어 있을 때만** 돌기 때문입니다.
스키마를 고치고 다시 띄우면 아무 일도 일어나지 않은 채 앱만 깨집니다.

> Spring Boot 4는 자동 구성을 기술별 모듈로 쪼갰습니다. `flyway-core`만 넣으면
> 라이브러리는 클래스패스에 있는데 자동 구성이 붙지 않아 **마이그레이션이 조용히 실행되지 않습니다.**
> 기동은 멀쩡히 성공하고 테이블만 없는 상태가 되므로 특히 알아채기 어렵습니다.
> `org.springframework.boot:spring-boot-flyway`가 필요합니다.

## 활성 슬롯 만료 라이프사이클

활성 슬롯의 만료는 단계마다 다시 찍힙니다. 값은 하나(`active` ZSet의 score)뿐이고,
그게 곧 "이 사람이 자원을 쥐고 있는 시간"입니다.

| 단계 | 만료 | 뜻 |
|---|---|---|
| 승격(스케줄러) | `admission-grace` 150초 | 입장권을 쓸 시간 |
| 입장 확정(claim) | `session-ttl` 10분 | 로그인할 시간 |
| 로그인 | `reservation-ttl` (설계 3분, 현재 1분) | 예약할 시간 |
| 로그아웃 | 즉시 삭제 | — |

만료 판정을 위한 인터셉터를 따로 두지 않은 이유가 여기 있습니다.
기존 게이트가 이미 매 요청 이 score를 보고 있어서, 로그인 시점에 값을 짧게 다시 찍는 것만으로
"제한 시간이 지나면 예약할 수 없다"가 그대로 강제됩니다. 진실 원천이 둘로 갈라지지 않습니다.

> 그래서 다음 페이즈에서 좌석·결제 엔드포인트를 만들 때 **`WebConfig`의 화이트리스트에
> 반드시 추가해야 합니다.** 게이트를 타지 않는 경로는 제한 시간이 지나도 그대로 열려 있습니다.

## 구현하면서 정한 것

**토큰은 JS 메모리에만 둡니다**

`sessionStorage`에 넣으면 새로고침에도 살아남아 취소 규칙과 어긋납니다.
메모리에 두면 새로고침으로 자연히 사라져 규칙과 맞습니다.
대신 대기 화면을 별도 페이지로 만들 수 없습니다 — 페이지를 이동하는 순간 토큰이 날아가므로,
랜딩 위에 뜨는 팝업으로 구현했습니다.

**팝업은 네이티브 `<dialog>`입니다**

포커스 트랩, Esc 처리, 배경 딤(`::backdrop`)을 직접 만들지 않아도 됩니다.
배경 클릭으로는 안 닫히는 게 기본 동작인데, 실수로 자리를 잃는 걸 막아 주므로 그대로 뒀습니다.

닫기 버튼은 `dialog.close()`만 부르고, 정리는 `close` 이벤트 한 곳에서 합니다.
버튼에만 정리를 붙이면 **Esc로 닫았을 때 폴링이 계속 돌아** 유령 요청이 남습니다.
만료(404) 경로도 같은 곳으로 모읍니다.

`showModal()`은 첫 포커스 가능 요소를 잡기 때문에, 그냥 두면 '닫기'에 포커스 링이 걸려
닫으라고 재촉하는 화면처럼 보입니다. 숨김 제목에 초기 포커스를 줘서 링을 없앴습니다.

**앞뒤 사람 수는 한 스냅샷에서 읽습니다**

`ZRANK`가 0-based이므로 `내 앞 = ZRANK(token)`, `내 뒤 = ZCARD - ZRANK - 1`입니다.
두 명령을 따로 부르면 그 사이에 진입이 끼어들어 합이 어긋나므로,
진입 때처럼 Lua로 묶었습니다([status.lua](../waiting-server/src/main/resources/redis/status.lua)).
`k6/status.js`가 매 응답마다 `앞 + 뒤 + 1 = 전체`를 검사합니다.

**windowId는 클라이언트가 돌려보냅니다**

서버 시계에서 다시 구하면, 자정 직전에 진입한 사용자가 자정 직후에 조회할 때
창이 달라져 자기 항목을 못 찾습니다. 키 TTL이 "창 마감 + 1시간"이라 그 사이 조회는 유효해야 합니다.
다만 이 값이 그대로 Redis 키가 되므로 `^\d{8}$`로 형식을 검사합니다.

**폴링은 `setInterval`이 아니라 `setTimeout` 체인입니다**

응답이 주기보다 느려지면 `setInterval`은 요청을 겹쳐 쌓습니다.
대기 인원이 많을수록 느려지는 구조라, 하필 느릴 때 부하를 스스로 키우게 됩니다.

**폴링에 지터를 더했습니다**

정각에 버튼이 열리는 시나리오에서는 수많은 사용자가 거의 같은 순간에 대기열
진입에 성공하고, 그 직후 첫 조회도 같은 위상으로 시작합니다. 이후 모두 같은
위상을 유지한 채 폴링을 반복하면 주기마다 요청이 파도처럼 몰립니다.

두 가지를 같이 했습니다. 먼저 **주기 자체를 2초에서 5초로 올렸습니다** —
지터는 몰림(분산)만 줄일 뿐 총 요청량은 그대로라, 대규모 동시 접속에서는 주기를
늘려 rps 자체를 줄이는 쪽이 더 큰 지렛대입니다. 그 위에 첫 폴링을
`0~POLL_INTERVAL_MS` 구간에서 무작위 지연 후 실행하고, 이후 매 주기에도 ±25%
지터(`POLL_JITTER_MS`)를 더해 위상이 다시 맞아들어가는 걸 막습니다. 에러
백오프도 배수 증가만 있으면 동시 장애 복구 시 재시도가 다시 몰리므로, "절반
고정 + 절반 무작위"로 흩었습니다. 순번 계산(진입 순서)에는 지터를 넣지
않았습니다 — 지터는 오직 "언제 상태를 물어보는가"에만 적용됩니다.

**클라이언트 타임아웃이 필요했습니다**

Redis를 내리고 버튼을 눌러 봤더니 서버가 503을 주는 게 아니라 그냥 매달렸습니다
(기본 프로파일의 Redis 타임아웃이 60초). 버튼이 잠긴 채 아무 안내도 안 나옵니다.
진입 10초, 조회 5초로 `AbortSignal.timeout`을 걸어 클라이언트가 먼저 끊게 했습니다.

**정원을 유지합니다. 속도를 정하지 않았습니다**

"초당 N명 방출"로 만들 수도 있었지만, 그건 유입만 늦출 뿐 활성 인원은 계속 쌓입니다.
정원을 기준으로 하면 뒷단(로그인·예약)이 동시에 상대할 인원에 상한선이 그어집니다.
대신 활성 인원의 이탈과 만료를 추적해야 해서, 나간 사람을 알아내는 일이 새로 생깁니다.

**활성 사용자를 ZSet에 담고 만료시각을 score에 넣습니다**

`active:{event}:{uuid}`처럼 키를 하나씩 두고 Redis TTL에 맡기면 만료는 공짜지만,
**인원을 세려면 `SCAN`이 필요합니다.** 정원을 유지하려면 매 주기 인원을 세야 하므로
그 비용을 낼 수 없습니다. 만료시각을 score에 넣으면 정리는 `ZREMRANGEBYSCORE` 한 번,
인원은 `ZCARD`로 정확히 나옵니다.

**승격은 스케줄러가 합니다. 조회 요청이 하지 않습니다**

조회 요청이 "내 앞에 자리가 있나"를 판정해 스스로 입장하게 만들 수도 있었습니다.
그러면 스케줄러도 리더 선출도 필요 없고, 폴링하지 않는 유령이 슬롯을 먹지도 않습니다.
그런데 만 명이 5초마다 폴링하면 약 2,000 rps인데, 그 경로에 쓰기를 섞는 대가가 큽니다.
승격을 초당 한 번 한 곳으로 모으면 조회는 읽기 전용으로 남습니다.

**입장권 유효시간이 두 단계입니다**

승격 직후에는 `admission-grace`(150초)만 주고, 사용자가 입장을 확정하면
`session-ttl`(10분)로 늘립니다. 유령은 폴링을 하지 않으니 확정까지 오지 못하고
150초 뒤 슬롯이 회수됩니다.

150초는 폴링 주기(5초)의 30배입니다. 브라우저가 비활성 탭의 타이머를 강하게 늦추기 때문에,
딴 탭에 가 있던 정상 사용자가 승격을 놓칠 여유를 준 값입니다.
짧게 잡을수록 유령 회수는 빨라지고 정상 사용자가 자리를 잃을 위험은 커집니다.

**한 주기에 올리는 인원에 상한을 뒀습니다**

Redis는 단일 스레드라 스크립트가 도는 동안 다른 모든 요청이 멈춥니다.
정원이 통째로 비는 순간 만 명을 한 번에 올리면 그게 그대로 지연 스파이크가 됩니다.
`max-batch`(500)로 잘라서 여러 주기에 나눠 올립니다.

**입장하면 대기 토큰을 쿠키로 바꿉니다**

대기 토큰은 JS 메모리에만 살아서 페이지를 옮기는 순간 사라집니다.
입장 확정 응답이 `pass` 쿠키(HttpOnly)를 심어 로그인·예약 화면까지 자격이 따라가게 합니다.
"새로고침하면 자리를 잃는다"는 규칙은 **대기 중에만** 적용됩니다 —
입장한 뒤에도 새로고침으로 자리를 잃는다면 그게 더 이상한 동작입니다.

쿠키에는 `Secure`를 붙이지 않았습니다. 로컬이 http라 붙이면 저장 자체가 안 됩니다.
HTTPS로 올리는 순간 반드시 켜야 합니다.

**Spring Security를 넣지 않고 BCrypt만 가져왔습니다**

`spring-security-crypto` 하나만 넣으면 필터 체인 없이 `BCryptPasswordEncoder`만 씁니다.
스타터를 넣으면 `permitAll`이어도 모든 요청이 필터 체인을 통과하는데,
이 프로젝트는 진입 처리량 수치의 변화를 근거로 삼기 때문에
**대기열 경로에 측정 변수를 하나 더 얹는 대가**가 큽니다.

인터셉터를 `/**` + exclude가 아니라 `/login`·`/reservation` 화이트리스트로 등록한 것도 같은 이유입니다.
대가는 새 화면을 만들 때마다 등록을 빠뜨리면 그대로 뚫린다는 점입니다.

**게이트는 쿠키의 존재가 아니라 Redis를 봅니다**

쿠키는 클라이언트가 들고 있는 값이라 얼마든지 지어낼 수 있습니다.
매 진입마다 `active` ZSet에 그 토큰이 살아 있는지 확인해야 정원과 만료가 실제로 강제됩니다.
`status.lua`가 이미 `ZSCORE`로 만료까지 판정하고 있어서 새 스크립트 없이 그대로 씁니다.
화면 진입당 한 번이라 폴링 경로와 달리 인원수만큼 곱해지지 않습니다.

그래서 **입장권 만료가 로그인 세션을 이깁니다.** 로그인한 채로 활성 자리가 회수되면
`/reservation`은 랜딩으로 돌려보냅니다. 정원 회수가 로그인 유지보다 우선입니다.

**세션 ID를 URL에 싣지 않습니다**

톰캣 기본값(`cookie,url`)이면 아직 쿠키를 돌려주지 않은 클라이언트에게
URL에 `;jsessionid=...`를 붙여 줍니다. 로그인 직후 리다이렉트가 딱 그 경우라,
세션 ID가 `Location` 헤더에 실려 브라우저 기록·Referer·접근 로그에 남습니다.
세션 ID는 곧 로그인 자격이므로 `tracking-modes: cookie`로 못 박았습니다.

세션 고정 방어(`changeSessionId()`)에는 함정이 하나 있습니다.
세션이 없을 때 부르면 `IllegalStateException`이 납니다.
이 앱은 로그인 전에 세션을 만드는 경로가 없어서, 검사 없이 부르면
**정상 로그인이 항상** 500으로 끝납니다.

## 남은 문제 — 유령 항목

대기열에서 빼는 수단이 없어서, 새로고침으로 재진입하면 이전 ZSet 항목이 남습니다.
그 유령들이 뒤에 오는 사람들의 "내 앞 사람 수"를 부풀립니다.
30만 건 테스트에서 `ZCARD`가 정확히 맞았던 만큼, 틀어지면 바로 드러납니다.

입장 제어가 붙으면서 대가가 하나 늘었습니다. 유령도 순서가 되면 승격되어
아무도 쓰지 않는 활성 슬롯을 `admission-grace`만큼 잡습니다.
순번이 부풀 뿐 아니라 **실효 정원까지 깎입니다.**

선택지는 대략 셋입니다.

- `DELETE /api/v1/waiting-queue/{token}` + 이탈 시 `sendBeacon` 호출
  — 확실하지만 beacon이 항상 도착한다는 보장은 없음
- 항목별 TTL + 주기적 하트비트 — 유실에 강하지만 조회마다 비용
- 유령을 그냥 두고 순번을 "최대치"로 안내 — 가장 단순, 정확도는 포기

폴링이 이미 5초마다 오고 있으므로, 그걸 하트비트로 재해석하는 두 번째 선택지가
새 요청을 만들지 않는다는 점에서 유리해 보입니다. 다만 조회 경로에 쓰기가 생기는데,
그건 승격을 스케줄러에 맡기며 피하려던 바로 그 비용입니다.

## 알려진 한계

**CSRF 토큰이 없습니다.** Spring Security를 도입하지 않았기 때문입니다.
세션 쿠키에 `SameSite=Lax`를 걸어 크로스 사이트 POST에 쿠키가 실리지 않게 하는 것으로
대신하고 있는데, 이건 토큰의 완전한 대체가 아닙니다. 같은 사이트 안의 XSS나
`SameSite`를 무시하는 오래된 클라이언트에는 소용이 없습니다.

**세션이 인스턴스 메모리에 있습니다.** 톰캣 기본 세션이라
