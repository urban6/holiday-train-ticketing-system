# 시연용 시드

> [← README](../README.md)

k6를 돌리지 않고 브라우저에서 대기 화면을 보려면 대기열에 사람이 있어야 합니다.
랜딩에 `?demo`를 붙이면 더미 대기자를 밀어넣는 패널이 우하단에 뜹니다.

```
http://localhost:8080/?demo
```

인원을 넣고 `채우기` → `명절 승차권` 클릭 → 팝업의 순번이 그 인원부터 시작해
초당 `queue.max-batch`(500)명씩 줄어들다가 입장합니다.

```bash
# 패널 없이 API로 직접 조작해도 됩니다
curl -X POST 'http://localhost:8080/api/v1/demo/queue?count=20000'   # → {"waiting":20000,"active":0}
curl      'http://localhost:8080/api/v1/demo/queue'                  # 현재 인원
curl -X DELETE 'http://localhost:8080/api/v1/demo/queue'             # 창 전체 초기화
```

**인원은 원하는 대기 시간 × 625로 잡습니다.** 대기열은 승격(`max-batch`, 초당 500명)과
봇 이탈(아래, 평균 초당 125명)로 빠지므로 체감 시간이 대략 `인원 ÷ 625초`입니다.
5,000명은 8초인데 폴링 주기가 5초라 순번이 한 번 찍히고 끝납니다.
기본값을 2만(약 32초)으로 둔 이유입니다.

**봇은 기다리다 포기하기도 합니다.** 승격 인원은 `promote.lua`가
`min(capacity - active, max-batch)`로 정하는데, 아래 회수 스케줄러가 정원을 계속 비워 주므로
매 주기 정확히 `max-batch`가 됩니다. 그래서 시드만 있으면 순번이 **초당 정확히 500씩** 줄어
기계처럼 보입니다. 승격 인원 계산은 프로덕션 경로라 건드리지 않고, 대신 `BotAttritionScheduler`가
1초마다 `0 ~ queue.demo.max-attrition`(기본 250)명을 대기열에서 무작위로 뺍니다.

```
max-attrition: 0            max-attrition: 250
-500  -500  -500  -500      -650  -520  -710  -660  -539
```

`ZRANDMEMBER`로 **큐 전체에서 흩어서** 뺍니다. 앞에서 순서대로 빼면 그건 이탈이 아니라 승격이고,
뒤에서만 빼면 내 순번이 줄지 않습니다. `demo-drain.lua`와 같은 `bot:` 접두사 필터를 쓰므로
**대기 중인 진짜 사용자는 이탈 대상이 아닙니다** — 무작위로 뽑는 경로라 필터가 깨지면 사용자가
소리 없이 큐에서 사라지므로, 테스트로 못 박아 두었습니다.
`max-attrition: 0`으로 두면 이탈이 꺼지고 정확히 `max-batch`씩 줄어듭니다.

**봇은 승격되는 즉시 슬롯을 반납합니다.** 이게 이 기능에서 유일하게 조심할 부분입니다.
봇도 진짜 사용자와 똑같이 `promote.lua`를 거치는데(그래야 순번이 줄어드는 속도가 사실과 같습니다),
봇은 폴링도 `claim`도 하지 않으므로 그냥 두면 아무도 쓰지 않는 활성 슬롯을
`admission-grace`(150초)만큼 붙잡습니다. [유령 항목](design.md#남은-문제--유령-항목) 문제 그 자체입니다.
2만 명을 시드하면 정원이 봇으로 가득 차서 **진짜 사용자가 한 시간 넘게 못 들어옵니다.**
그래서 `BotDrainScheduler`가 200ms마다 `active`에서 봇만 골라 `ZREM`합니다 —
로그아웃이 슬롯을 반납하는 것과 같은 동작이라 "입장하자마자 나가는 사용자"인 셈입니다.

> 주기가 `promote-interval`(1초)보다 짧은 건 승격 직후 봇이 정원을 채운 상태로 다음 승격이
> 돌면 `room <= 0`이라 그 주기가 통째로 헛돌기 때문입니다. 대기열이 빠지는 속도가 절반이 됩니다.

알아 둘 것:

- **`전체 비우기`는 진행 중인 실사용자도 끊습니다.** `active`를 통째로 지우므로
  그 순간 로그인·예약 화면에 있던 사람은 다음 요청에서 게이트에 걸려 랜딩으로 돌아갑니다.
- **봇은 순번만 만듭니다.** 로그인도 예약도 하지 않으므로 DB·세션 쪽 부하는 재현되지 않습니다.
  그쪽까지 필요하면 k6가 맞는 도구입니다.
- **봇 이탈은 시연용 장식입니다.** 다음 페이즈의 대기열 이탈 처리를 미리 흉내 낸 것일 뿐,
  실사용자의 이탈은 여전히 처리되지 않습니다([유령 항목](design.md#남은-문제--유령-항목)).

**규모를 키울 때.** `max-count`는 100만입니다. 실측한 비용은 아래와 같습니다.

| 인원 | 메모리 | 시드 시간 | 순번 조회 | 다 빠지는 데 |
|---|---|---|---|---|
| 10만 | 6.6MB | 0.4초 | 3~10ms | 2.7분 |
| 100만 | 66MB (상한의 26%) | 2.8초 | 3~10ms | **26분** |

메모리도 시드 속도도 조회 성능도 제약이 아닙니다. 봇 항목은 69바이트라
(`bot:12345`가 36자 UUID보다 짧아 실사용자의 115바이트보다 작습니다)
`maxmemory 256MB`의 물리적 한계는 약 380만 건이고, `active` 키와 작업 메모리를 빼면
실질 300만 건쯤입니다. 순번 조회는 `ZRANK`가 O(log N)이라 인원수를 타지 않습니다.

**걸리는 건 감소 속도뿐입니다.** 승격 500 + 이탈 평균 125 = 초당 약 625명이라
100만 명은 26분입니다. 큰 규모를 짧게 보려면 이탈을 함께 올리세요.

```bash
# 100만 명을 약 90초에 빠지게
java -jar waiting-server/build/libs/waiting-0.0.1.jar --queue.demo.max-attrition=20000
```
- 시드는 창(`windowId`) 단위입니다. KST 자정이 지나면 새 창이라 다시 채워야 합니다.
- 승차권 버튼을 **먼저** 누른 뒤 시드하면 봇이 내 뒤에 붙습니다. "내 뒤의 대기자 수"를
  채우고 싶을 때 쓰면 됩니다.

**이 기능은 `queue.demo.enabled`로 통째로 꺼집니다.** 끄면 빈이 하나도 등록되지 않아
API가 404가 되고 회수 스케줄러도 돌지 않으며 `?demo`를 붙여도 패널이 렌더링되지 않습니다.
인증이 없어 켜져 있으면 누구나 대기열을 부풀릴 수 있으므로 **외부에 노출되는 환경에서는
반드시 false여야 합니다.** `loadtest` 프로파일에서는 이미 꺼 두었습니다 —
켜 두면 회수 스케줄러가 200ms마다 `active`를 훑어서 측정 대상이 대기열이 아니라 봇 청소가 됩니다.
