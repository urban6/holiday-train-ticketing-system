# 측정

> [← README](../README.md)

부하 도구는 k6입니다. 시나리오가 둘입니다.

```bash
# 앱을 loadtest 프로파일로 띄운 뒤

# 진입: N명분 요청을 최대한 빠르게 밀어넣는다
k6 run -e USERS=100000 k6/enqueue.js

# 조회: N명이 대기하면서 5초 간격(지터 포함)으로 자기 순번을 폴링한다
k6 run -e WAITERS=5000 k6/status.js
```

두 수치를 같은 축에 놓고 비교하면 안 됩니다.
진입은 1인당 1회지만 조회는 1인당 N회라, 초당 조회 수가
`대기 인원 ÷ 폴링 주기`만큼 곱해집니다.

앱은 **IDE Run 버튼이 아니라 빌드된 jar로** 띄웁니다.
IDE는 `-XX:TieredStopAtLevel=1`을 붙이는데, C2 컴파일러를 꺼서
아무리 워밍업해도 최적화된 코드에 도달하지 못하기 때문입니다.

```bash
cd waiting-server && ./gradlew bootJar && cd ..
java -jar waiting-server/build/libs/waiting-0.0.1.jar \
     --spring.profiles.active=loadtest
```

`loadtest` 프로파일은 측정을 왜곡하는 것만 끕니다.
특히 요청당 `log.debug`를 끄는 게 중요한데, `System.out`의 락이
가상 스레드에서 pinning 지점이라 켜두면 대기열이 아니라 로거를 측정하게 됩니다.

큐에 실제로 쌓였는지, 그리고 다시 돌리기 전 초기화:

```bash
W=$(TZ=Asia/Seoul date +%Y%m%d)
docker exec redis redis-cli ZCARD waiting:holiday:$W   # 대기 인원
docker exec redis redis-cli ZCARD active:holiday:$W    # 활성 인원 (capacity를 넘으면 안 됨)

# 초기화. active 키까지 지워야 하므로 패턴이 'waiting:'이 아니라 '*:holiday:*'다.
docker exec redis redis-cli --scan --pattern '*:holiday:*' \
  | xargs -n1 docker exec redis redis-cli UNLINK
```

키 TTL이 "내일 자정 + 1시간"이라 초기화하지 않으면 같은 날 계속 누적됩니다.

정원을 바꿔 가며 보려면 실행 인자로 덮어씁니다.

```bash
java -jar waiting-server/build/libs/waiting-0.0.1.jar \
     --spring.profiles.active=loadtest \
     --queue.capacity=1000 --queue.max-batch=500 --queue.admission-grace=150s
```

## 환경

k6·애플리케이션·Redis가 **모두 같은 Mac에서** 돕니다.
부하 생성기와 측정 대상이 CPU를 나눠 쓰므로 **절대 수치는 실제 서버 성능이 아닙니다.**
목적은 절대값이 아니라 **병목이 어디서 어떤 모양으로 나타나는가**와
**변경 전후의 상대 비교**입니다.

Redis는 컨테이너 CPU 1코어, `maxmemory 256mb` + `noeviction`입니다.
대기열 항목 하나가 약 115바이트라 **약 220만 건**에서 쓰기가 거부되기 시작합니다.
버그가 아니라 설계된 동작입니다.

## 결과

2026-07-22. `loadtest` 프로파일 jar, 요청당 로그 제거,
`kern.ipc.somaxconn=1024` + `server.tomcat.accept-count=1000`.
각 런 30만 건, 매 런 전 Redis 초기화.

| 동시 실행(VUS) | 실효 rps | avg | p95 | 실패 |
|---|---|---|---|---|
| 500 | 33,073 | 14.86ms | 27.64ms | 0 |
| 1,000 | **34,129** | 28.73ms | 51.64ms | 0 |
| 2,000 | 30,912 | 61.80ms | 111.68ms | 0 |
| 4,000 | 29,096 | 131.63ms | 250.44ms | 0 |

동시 실행을 2배로 늘릴 때마다 **응답 시간은 정확히 2배가 되고 처리량은 제자리**입니다.
창구 수가 정해진 은행에 손님만 늘리는 것과 같아서, 줄이 길어질 뿐 처리량은 안 늘어납니다
(`처리량 = 동시성 ÷ 응답시간`). 이 시스템의 최적 지점은 VUS 500~1,000 근처입니다.

네 런 모두 `ZCARD`가 정확히 300,000이었습니다.
초당 3만 건씩 밀어넣는 동안 순번 유실도 중복도 없었습니다 —
`INCR`과 `ZADD`를 Lua로 원자적으로 묶은 것이 의도대로 동작했습니다.

## 입장 제어 검증

2026-07-23. `capacity=1000`, `max-batch=500`, `promote-interval=1s`, `admission-grace=60s`.
만 명을 밀어넣고 2,000명이 90초간 폴링했습니다.

- **정원을 넘지 않았습니다.** 0.5초 간격으로 `ZCARD active`를 200회 관측하는 동안
  최대치가 정확히 1,000이었습니다. `waiting + active = 10,000`도 유지됐습니다.
- **폴링 18만 건 전부 성공**했고 `앞 + 뒤 + 1 = 전체`가 한 번도 어긋나지 않았습니다.
  조회 p95는 58.51ms(임계 500ms).
- **`max-batch`가 실제로 관측됩니다.** 첫 1,000명의 입장권이 만료돼 슬롯이 한꺼번에
  회수된 순간, 대기 인원이 `10,000 → 9,500 → 9,000`으로 **두 주기에 나눠** 줄었습니다.
  상한이 없었다면 1,000명이 한 스크립트에서 처리됐을 자리입니다.

배치가 도는 순간의 지연 스파이크는 이 해상도(1초 간격 관측)로는 분리해 내지 못했습니다.
`max-batch` 값을 근거 있게 정하려면 배치 실행 시각과 요청 지연을 같은 축에 놓고 봐야 합니다.

Redis를 내리고 확인한 것 — 스케줄러가 죽지 않고 주기마다 경고 한 줄만 남기며,
Redis를 되살리면 다음 주기에 스스로 재개합니다. 진입 API는 그대로 503입니다.

> 이 런은 Docker 데몬이 꺼져 있어 **호스트의 Redis**(Homebrew, 포트 6380)로 측정했습니다.
> 위의 진입 처리량 표와 절대값을 나란히 두면 안 됩니다 — 병목의 정체인
> Docker 왕복 비용이 빠져 있습니다.

## 로그인 도입 후 회귀 확인

2026-07-23. 게이트(인터셉터)가 붙으면서 진입 경로가 느려졌는지 확인했습니다.
같은 머신에서 변경 전(HEAD) jar와 변경 후 jar를 **번갈아 3회씩** 돌렸습니다.
각 런 15만 건, VUS 1,000, 매 런 전 워밍업 300건 + Redis 초기화.

| | r1 | r2 | r3 | 평균 |
|---|---|---|---|---|
| 변경 전 | 18,378 | 18,036 | 20,712 | 19,042 |
| 변경 후 | 14,864 | 20,665 | 19,340 | 18,290 |

**측정 가능한 차이가 없습니다.** 두 계열의 산포(14.9k~20.7k)가 평균 차이(4%)보다 훨씬 큽니다.
인터셉터를 `/**` + exclude가 아니라 `/login`·`/reservation` 화이트리스트로 등록해서,
대기열 API는 인터셉터 자체를 타지 않습니다.

**다만 이 표의 절대값을 위의 34,129 rps와 나란히 두면 안 됩니다.**
변경 전 jar조차 오늘은 19k 언저리로 나왔습니다. 같은 코드가 그날 34k, 오늘 19k라면
차이는 코드가 아니라 머신 상태입니다. **비교는 같은 세션 안에서만** 유효합니다.

## 병목

애플리케이션을 빼고 `redis-benchmark`로 같은 Lua 스크립트를 직접 때려 비교했습니다
([k6/redis-baseline.sh](../k6/redis-baseline.sh)).

| 측정 대상 | 처리량 |
|---|---|
| Redis 자체 (컨테이너 안) | 107,009 rps |
| 호스트 → Docker → Redis (앱과 같은 경로) | 37,908 rps |
| Spring Boot 앱 전체 | 34,129 rps |

**병목은 Spring Boot가 아니라 Docker 네트워킹입니다.**
앱의 34k는 호스트에서 Redis에 직접 접속했을 때의 한계인 38k에 거의 붙어 있고,
Spring Boot가 추가로 잡아먹은 건 10% 정도입니다.
반면 컨테이너 안팎에서 3배 가까이 깎입니다.

요청당 Docker 왕복 비용은 약 0.31ms입니다
(호스트 p50 0.399ms − 컨테이너 p50 0.087ms).
Redis가 실제로 일하는 시간은 **8.37μs**이므로,
**일하는 시간의 약 37배를 오가는 데** 쓰고 있는 셈입니다.

macOS에서 Docker가 리눅스 VM 안에서 도는 구조 때문이며 **리눅스 서버에는 없는 비용**입니다.
따라서 이 환경에서는 코드나 Redis 설정을 최적화해도 34k 위로 올라가지 않습니다.
그 위를 보려면 Redis를 리눅스 머신에 올리거나 부하 생성기를 분리해야 합니다.
